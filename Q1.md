# 80TB 디자인 에셋 AI 관리 시스템 — 초기 기획 검토 (Q1)

꼼꼼히 읽어봤습니다. 제안서 자체는 큰 그림에서 타당하지만, 실제 80TB를 다루려면 **몇 군데는 반드시 재검토**해야 합니다. PM·아키텍트와 킥오프 미팅을 한다는 가정으로, 질문-의견-결정사항 순서로 정리하겠습니다.

---

## Ⅰ. 먼저 확인해야 할 전제 (모든 설계의 근거)

### Q1. "80TB"의 실체 — 파일 수와 구성은요?

80TB는 단순 용량일 뿐, **파일 수가 설계를 좌우**합니다.

| 평균 파일 크기 가정 | 예상 파일 수 | 시사점 |
|---|---|---|
| 50MB (PSD 중심) | ~160만 개 | pgvector HNSW 인덱스 감당 가능 |
| 200MB (대형 PSD + RAW) | ~40만 개 | 여유 있음 |
| 5MB (JPG/PNG 섞임) | ~1,600만 개 | **pgvector 한계, Qdrant/Milvus 검토 필요** |

**확정 필요:**
- [ ] 파일 확장자별 분포(PSD, AI, INDD, JPG, PNG, PDF, MP4 등) 대략적 비율
- [ ] 현재 NAS 종류(Synology/QNAP/자체 구축) 및 네트워크(1G/10G)
- [ ] 연간 증가량 — 이게 전처리 파이프라인의 지속 부하를 결정합니다

---

### Q2. 디자이너 3인의 **현재 워크플로우**는?

시스템을 기획할 때 가장 흔한 실패 원인이 "만들어 놨는데 안 씀"입니다.

- 현재 파일은 어떻게 찾고 있나요? (폴더 구조 / 파일명 규칙 / 기억)
- 가장 큰 페인포인트는? ("옛날에 만든 비슷한 시안 찾기 어려움" vs "중복 파일 난립" vs "누가 뭐 만들었는지 모름")
- 디자이너가 **업로드 주체**인가요, 아니면 기존 NAS의 변경을 **자동 감지**해야 하나요?

**제 추천:** 첫 번째 페인포인트를 한 개만 선택해 Phase 1에서 완벽 해결 → 신뢰 확보 후 기능 확장. 욕심내서 기능을 다 넣으면 이런 프로젝트는 대부분 90% 완성 상태로 멈춥니다.

## Ⅱ. 제안 아키텍처에 대한 비판적 검토

### A. PostgreSQL + pgvector — **유지, 단 조건부**

좋은 선택입니다. 단 다음 기준을 지키실 때만요.

- 벡터 수 **500만 개 이하** → pgvector HNSW로 충분
- 500만 ~ 2,000만 → pgvector 여전히 가능하나 튜닝 난이도 상승
- 2,000만 초과 → **Qdrant 또는 Milvus로 분리** 권장 (메타는 PG, 벡터는 전용 DB)

Q1의 파일 수 파악이 이 결정의 핵심입니다.

### B. n8n — **부분 동의, 역할 축소 필요**

n8n은 **오케스트레이션용으로만** 쓰시고, 실제 AI 인퍼런스는 별도 워커로 빼시길 권합니다.

**이유:**
- n8n 노드 내부에서 CLIP을 돌리면 워커 재시작/큐 관리/재시도 제어가 번거롭습니다
- 대량 배치(수십만~수백만 이미지)에서 n8n의 실행 로그·큐가 폭주합니다

**구성 제안:**
```
n8n (트리거, 상태 관리, 알림)
  → Redis/RabbitMQ 큐
    → Python 워커 (CLIP, OCR, 썸네일) ×N대
      → PostgreSQL + pgvector
```

Python 워커는 **Celery / Dramatiq / RQ** 중 선택. 이 구조면 GPU 자원 스케일링이 쉽고, n8n은 "이 폴더 스캔 → 큐에 넣기 → 완료 알림" 수준의 가벼운 역할만 맡게 됩니다.

### C. ResourceSpace — **재검토 권장**

오픈소스 DAM 중 성숙도는 높지만, 2026년 현재 기준:
- PHP 기반, UX가 다소 올드
- 벡터 검색을 네이티브로 지원하지 않아 커스텀 모듈을 상당량 작성해야 함
- 디자이너 3인 소규모 팀이라면 오버엔지니어링일 수 있음

**대안 3가지:**

| 옵션 | 공수 | 장점 | 단점 |
|---|---|---|---|
| ResourceSpace 커스터마이징 | 중 | 권한/버전 관리 기본 제공 | 벡터 검색 붙이기 번거로움, UX 구식 |
| **Immich (사진 DAM) 포크** | 중 | 모던 UI, CLIP 내장, 썸네일 파이프라인 완비 | 디자인 파일(PSD/AI) 지원은 직접 추가 |
| 자체 구축 (Next.js + FastAPI) | 중~상 | 100% 요구사항 반영 | 초기 개발 공수 큼 |

**제 추천:** **Immich를 베이스로 포크** — CLIP 임베딩, HNSW 검색, 썸네일 생성, 권한 관리가 이미 구현되어 있어 80% 이상 재활용 가능합니다. PSD/AI 확장자 핸들러만 추가하면 됩니다.

## Ⅲ. 제안서에서 **빠진 핵심 이슈들**

### ① 썸네일 파이프라인의 기술적 난이도

"썸네일 추출"이 한 줄로 적혀 있는데, 실제론 확장자별로 전혀 다른 문제입니다.

| 확장자 | 도구 | 난이도 |
|---|---|---|
| PSD | `psd-tools` (Python) 또는 ImageMagick | 쉬움 — 단 레이어 수천 개짜리 대형 파일은 수십 초 소요 |
| AI (Illustrator) | Ghostscript 또는 `pdftoppm` (AI는 PDF 호환) | 중간 — 구버전 AI는 실패율 있음 |
| INDD (InDesign) | **상용 라이선스 필요 또는 추출 불가** | **높음 — 별도 대응책 필요** |
| Figma | API 호출 (파일이 클라우드면) | 환경 의존 |
| Sketch | `sketchtool` (macOS 전용) | 리눅스 서버면 제약 |
| TIFF/RAW | ImageMagick + LibRaw | 쉬움 |

**확정 필요:** INDD 파일 비중이 높다면 별도 전략 필요. 예: 디자이너가 최종 단계에서 PDF export 의무화.

### ② 증분 처리(Incremental) — 프로젝트 지속성의 핵심

초기 80TB 인덱싱은 한 번이면 끝이지만, 매일 새 파일이 추가됩니다.

- 파일 변경 감지: **inotify** (리눅스) 또는 NAS 제공 이벤트 / 주기적 스캔
- 해시 기반 변경 감지: SHA-256 또는 더 빠른 xxHash
- 재처리 큐: 수정된 파일만 재임베딩

이 부분이 Phase 5로 따로 있어야 합니다. **"배포 후 끝"이 아니라 "배포 후 운영"이 진짜 시작**입니다.

### ③ 중복 제거 — 두 단계로 분리

제안서에 "중복 제거"가 한 줄인데, 실제로는 두 종류입니다.

1. **완전 동일 중복** — SHA-256 해시 매칭. CLIP 쓰지 말고 해시로 처리(훨씬 빠르고 정확).
2. **시각적 유사 중복** — CLIP 임베딩 코사인 유사도 > 0.95 등 임계값 기반.

시각적 중복은 "지울까 말까"를 **사람이 판단**해야 합니다. 자동 삭제는 절대 금물. 플래그만 달아 디자이너에게 리뷰 UI로 제공.

### ④ OCR — 한글 지원 명시 필요

CLIP은 영어 중심이고, 디자인 카피는 대부분 한글입니다.

- **추천:** PaddleOCR (한글 지원 양호, GPU 가속) 또는 EasyOCR
- 또는 **한국어 CLIP 대안:** KoCLIP, 또는 다국어 모델인 `openai/clip-vit-large-patch14` 대신 **`laion/CLIP-ViT-H-14-laion2B-s32B-b79K`** (다국어 대응 우수) 검토

### ⑤ GPU 하드웨어 — 아직 불명확

80TB 인덱싱에 필요한 GPU는:
- RTX 4090 1장 기준 CLIP ViT-L/14 배치 64로 **초당 200~400장**
- 200만 장 가정 시 이론상 **3~6시간**, 실제론 I/O 병목으로 **1~2일**
- RTX 4090 vs A6000 vs 중고 3090 2장 → 예산별 의사결정 필요

**확정 필요:** GPU 예산 범위 / 기존 보유 장비 / 서버 상면 유무

### ⑥ 백업 전략 — 80TB의 함정

원본 NAS는 백업되고 있으신가요? 인덱스 DB(PG)는?
- DAM 시스템이 원본 파일을 **이동/변경하지 않도록** 읽기 전용 마운트 권장
- PG + 벡터 인덱스 재구축 스크립트를 상시 갖춰 두면 DB 손실 시 원본에서 재복구 가능

### ⑦ 보안 / 접근제어

"로컬이라 안전"은 절반만 맞습니다.

- 디자이너 계정별 권한(본인 작업물만 vs 전체 열람)
- 외부(클라이언트) 공유가 필요한가요? 필요하면 임시 링크(TTL 24시간) 설계 필요
- 감사 로그(누가 언제 무엇을 다운로드) — 에셋 유출 추적용

## Ⅳ. 재정리한 로드맵 (Phase 구체화)

제안서 4단계를 **실제 실행 가능한 수준**으로 쪼개겠습니다.

### Phase 0 — 발견 (1~2주, 코드 없음)

- 파일 확장자 분포 조사 (`find` + 집계 스크립트)
- 디자이너 3인 각 1시간 인터뷰 → 페인포인트 Top 3
- GPU/서버 하드웨어 확정
- **산출물:** 인프라 사양서, 우선순위 요구사항 목록

### Phase 1 — 메타데이터 적재 (2~3주)

- PG 스키마 설계 (files, projects, tags, users, embeddings)
- 전체 파일 스캐너: 경로/크기/mtime/해시 → PG
- **중복 탐지 1차** (해시 기반) — 이것만으로도 수 TB 절감 가능성 있음
- **산출물:** 기본 검색 UI (파일명·확장자·기간 필터)

### Phase 2 — 썸네일 파이프라인 (2~3주)

- 확장자별 워커 구현 (PSD/AI/이미지/동영상)
- 큐 + 재시도 + 실패 로그
- **실패 파일 리포트**: "처리 불가 47개 파일" → 디자이너 대응
- **산출물:** 모든 파일에 썸네일 연결된 갤러리 뷰

### Phase 3 — AI 인덱싱 PoC (3~4주)

- **최근 1년치 또는 핵심 프로젝트 폴더**만 대상
- CLIP 임베딩 + PaddleOCR 병행
- pgvector HNSW 인덱스 빌드 후 실제 유사도 검색 품질 검증
- **검수 세션**: 디자이너와 함께 "비슷한 시안 찾아줘" 20케이스 테스트
- **산출물:** PoC 품질 리포트 → 전체 확장 여부 결정

### Phase 4 — 전체 확장 + 자동 태깅 (4~6주)

- PoC 검증 후 전체 80TB 인덱싱 (백그라운드 야간 실행)
- CLIP zero-shot 태깅 (계절/용도/분위기 등 사전 정의 태그셋 기반)
- 디자이너가 태그 수정·승인 UI

### Phase 5 — 운영 체제 (지속)

- 실시간 감시(inotify) 기반 증분 인덱싱
- 모니터링 대시보드 (처리 지연, 실패율, 디스크 사용량)
- 분기별 리뷰로 기능 개선

   - [ ] 현재 NAS 종류 + 네트워크 속도

4. **UI 방향성**

   - [ ] 외부 공유 요구 있음/없음

6. **마일스톤**
   - [ ] 언제까지 **첫 가시적 성과**(Phase 1 수준)가 필요하신가요?

## Ⅵ. 1차 답변 반영 및 확정 사항 (2026-04-19)

### 접수된 정보

- **파일 구성:** PSD 중심 + INDD 혼재. 동영상/3D는 확인 필요(추후 Phase 0에서 스캔으로 파악)
- **PoC 전략:** 서브디렉토리를 로컬에 복사해 테스트 → 전체 확장 (확정)
- **접근 방식:** 현재 네트워크 드라이브. 최종 목표는 웹 기반 Viewing + 업로드 + 수정

### 결정 1. 파일 재구조화 vs 링크 오버레이

**결론: 하이브리드(C안) — "원본은 그대로, 논리 레이어로 관리"**

**이유:**
- PSD의 Placed Smart Object, INDD의 링크된 이미지/폰트, AI의 Placed Image는 경로 변경 시 전부 **Missing Link** 오류가 발생
- 80TB 물리 이동은 다운타임·데이터 손실·링크 깨짐의 삼중 리스크
- 원본을 건드리지 않으면 DAM 장애가 발생해도 기존 워크플로우는 100% 유지됨

**실행 방식:**
1. **단계 1:** DAM은 **읽기 전용**으로 스캔·인덱싱만 수행. 디자이너는 평소대로 작업
2. **단계 2:** DAM 안에서 **가상 컬렉션/프로젝트 그룹**으로 논리적 재분류 (물리 이동 없음)
3. **단계 3 (6개월+):** 신규 프로젝트에만 새 명명 규칙 적용. 구 데이터는 검색으로 커버

**불문율:** "원본 NAS는 건드리지 않는다"

### 결정 2. PoC 로컬 샘플 기준

- **크기:** 100~500GB (하루 안에 재처리 가능한 규모)
- **구성:** PSD 다수 + INDD 소수 + 완결 프로젝트 1~2개 + 진행중 1개
- **복사 규칙:**
  - 서브디렉토리 **통째로** 복사 (상대 경로 링크 유지 → INDD 링크 문제 사전 검증)
  - `rsync -avh --progress` 사용 (중단/재개 가능)
  - 한글 파일명 인코딩 확인 (UTF-8 ↔ CP949 이슈 주의)
  - 원본·복사본 **해시 검증** 후 테스트 시작

### 결정 3. 접근 방식 진화 경로

```
[현재] 디자이너 PC → SMB/CIFS → NAS (80TB 원본)

[PoC]
디자이너 PC → SMB → NAS (읽기전용)
                ↓ 스캔
        DAM 서버 (PG + pgvector + 워커)
                ↓
       웹 UI (검색·갤러리)

[운영]
웹 UI → FastAPI → NAS (쓰기 허용 영역)
         ↓
     자동 인덱싱 파이프라인
```

**"수정" 기능에 대한 주의:**
- 메타데이터/태그 편집은 용이함
- **파일 내용 편집(실제 PSD 수정)은 브라우저에서 불가능** — Photopea 임베드가 한계
- 표준 패턴은 **"웹에서 다운로드 → 로컬 Photoshop 편집 → 웹에 업로드"** 루프
- 업로드된 신규 파일은 자동으로 인덱싱 파이프라인에 진입

### WSL2 환경에서 반드시 지켜야 할 사항

- **PoC 데이터는 WSL2 네이티브 파일시스템(ext4)에 배치** — `/mnt/c/...` 에 두면 I/O 속도가 10~100배 저하됨
- 권장 배치:
  ```
  /home/ktalpha/Work/Dam/          # 코드·문서·설정 (git 관리)
  /home/ktalpha/Work/Dam-data/     # PoC 실제 에셋 (git ignore, 100~500GB)
  /home/ktalpha/Work/Dam-db/       # PostgreSQL 데이터 볼륨 (별도)
  ```
- Docker Desktop + WSL2 + NVIDIA GPU 패스스루는 CUDA on WSL 드라이버 필요
- 소스 NAS가 Windows에서 네트워크 드라이브로 연결되어 있다면, WSL에서 접근 시 `/mnt/<drive>/...` 경로로 보임 (성능 병목 가능 → 초기 복사는 PowerShell `robocopy` 또는 Windows 탐색기 사용 권장)
